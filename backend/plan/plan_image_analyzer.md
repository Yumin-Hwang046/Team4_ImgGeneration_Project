사용자가 던진 사진을 컴퓨터(API)가 볼 수 있는 형태로 변환해서 OpenAI에 보낸 뒤, "이건 광택이 나는 하얀 컵에 담긴 커피야"라는 형태의 텍스트 답변을 돌려받게 만듭니다.

### 🛠️ 세부 개발 진행 순서 (할 일 목록)

| 상태 | 순서 | 목표 | 세부 내용 |
|:---:|:---:|---|---|

| [O] | 2 | **라이브러리 설치** | 통신을 위한 `openai` 툴과 이미지 처리를 위한 `Pillow` 등 필요 패키지 설치 (`pip install`) |

| [O] | 4 | **메인 통신 로직 작성** | GPT에게 내릴 프롬프트 명령을 작성하고, 변환된 사진을 넘겨 텍스트 응답을 받아오는 메인 함수(`analyze_image`) 완성 |
| [O] | 5 | **독립 테스트 진행 🔥** | 복잡한 웹 서버 구동 없이, `analyzer.py` 스크립트를 단독 실행하여 터미널에 요약 글이 나오는지 바로 체크! |

## 3. 작업 요약 및 트러블슈팅 (시행착오)

본 시각 분석기 모듈을 개발하면서 직면했던 주요 단계를 시간순으로 나열하고, 각 단계마다 어떤 방식과 문제를 거쳐왔는지 상세히 기록합니다.

### 📍 1단계: 키(Key) 관리 세팅
* **작업 내용:** OpenAI API 키를 발급받고, 프로젝트 폴더 최상단에 `.env` 파일을 생성하여 안전하게 저장했습니다.
* **도입 이유:** 외부(깃허브 등)에 비밀번호(API 키)가 날것으로 노출되는 치명적인 보안 사고를 방지하고 시스템 환경 변수로 깔끔하게 불러오기 위해서입니다.
* **발생한 문제:** 1단계 세팅 과정 자체에서는 별다른 에러가 발생하지 않았습니다.
* **해결 방법:** -

### 📍 2단계: 라이브러리 설치
* **작업 내용:** 서버 통신을 위한 `openai`, 환경 변수 관리를 위한 `python-dotenv`, 이미지 전처리를 위한 `Pillow` 패키지를 설치했습니다.
* **도입 이유:** 복잡한 로직(인터넷 통신, 환경 세팅, 이미지 압축)을 처음부터 직접 구현하지 않고, 검증된 도구를 활용해 개발 속도와 안정성을 극대화하기 위해서입니다.
* **발생한 문제:** 설치 시 충돌은 없었으나, 뒤이은 과정에서 마주칠 고화질 이미지의 용량 한계(통신 지연)를 극복하기 위해 `Pillow` 라이브러리의 선제적인 도입이 권장되었습니다.
* **해결 방법:** -

### 📍 3단계: 사진 변환기 제작 (`encode_image`)
* **작업 내용:** `Pillow`를 통해 원본 사진 해상도를 1024픽셀 이하로 압축(quality=85)한 뒤, `Base64` 문자열로 치환하도록 `encode_image` 함수를 작성했습니다.
* **도입 이유:** GPT-4o Vision 모델이 이미지를 시각적으로 인지하게 만들려면, 이미지 파일 자체가 아니라 구조화된 텍스트(Base64)로 변환해 네트워크로 전송해야 하기 때문입니다.
* **발생한 문제:** **[무한 로딩 통신 시간 초과 (Request Timed Out)]** 7.8MB짜리 고화질 사진을 압축 작업 없이 전체 문서를 Base64로 변환했더니, 텍스트 길이가 수천만 자를 초과하여 전송 단계에서 과부하가 발생했습니다.
* **해결 방법:** 전송 직전에 `Pillow`의 리사이징(`.thumbnail()`)과 압축 옵션을 활용해 이미지 용량을 경량화함으로써 구조적인 타임아웃 붕괴를 원천 차단했습니다.

### 📍 4단계: 메인 통신 로직 작성 (`analyze_image`)
* **작업 내용:** 변환된 Base64 텍스트 사진과 지시문(프롬프트)을 모델로 보내어 응답을 받는 `client.chat.completions.create` 메인 함수를 완성했습니다.
* **도입 이유:** 후속 모듈인 <광고 문구 생성기>는 이미지를 볼 수 없고 오직 텍스트만 처리할 수 있습니다. 이미지를 "글자로 쓰인 상품 정보"로 통역해 주기 위한 핵심 과정입니다.
* **발생한 문제:** 
  1. **[HTTP 403 에러]** `gpt-4o` 모델은 계정 사용 권한이 막혀있었습니다.
  2. **[HTTP 400 에러]** 모델을 `gpt-5-mini`로 교체했으나, 기존 파라미터(`max_tokens`)가 최신 토큰 규격과 맞지 않아 반려당했습니다.
* **해결 방법:** 목표 모델을 `gpt-5-mini`로 확정 변경하고, 파라미터를 시대에 맞게 `max_completion_tokens` 필드명으로 수정해 서버 접속망을 관통했습니다.

### 📍 5단계: 독립 테스트 진행 🔥
* **작업 내용:** 방대한 통합 웹 서버를 켜기 전에, `analyzer.py` 파일만 터미널에서 독자 실행되도록 테스트 코드를 심어 결과를 즉각 확인했습니다.
* **도입 이유:** 파이프라인의 개별 조각(모듈)이 잘 돌아가는지 미리 검증해야만, 나중에 합쳤을 때 어디서 에러가 났는지 정확히 색출할 수 있기 때문입니다.
* **발생한 문제:** **[응답 텍스트 빈 값(Blank) 현상]** 통신은 정상적으로 성공(200)했으나, AI 모델이 적절한 텍스트 응답을 출력하지 않고 완전히 빈 값을 반환하는 현상이 발생했습니다.
* **해결 방법:** `gpt-5-mini`와 같은 최신 추론형(Reasoning) AI가 대답을 하기 전 '생각'을 하다가 `max_completion_tokens=300`의 한도 용량을 허공에 다 소비하여 답변이 잘린 것이 원인이었습니다. 해당 한도를 삭제하여 AI가 충분히 추론 후 온전히 텍스트를 출력하도록 로직을 수정했습니다.
